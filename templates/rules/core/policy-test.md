# Policy Test

비즈니스 정책이 포함된 코드를 수정할 때 반드시 테스트로 보호한다.

## 정책 키워드

다음 키워드가 변경 범위에 포함되면 정책 코드로 간주한다:
- 날짜/시간 계산 — 만료일, 유효기간, 기한, D-day
- 금액/수량 계산 — 할인율, 수수료, 한도, 최소/최대값
- 상태 전이 — 승인/거부, 활성/비활성, 결제 상태
- 필터/정렬 조건 — 권한 체크, 노출 조건, 정렬 기준
- 외부 연동 규격 — API 요청/응답 포맷, 프로토콜 버전

## 필수 조치

1. **테스트 존재 확인** — 해당 정책을 검증하는 테스트가 있는지 확인한다.
2. **테스트 부재 시 작성** — 정책 로직의 경계값과 예외 케이스를 커버하는 테스트를 먼저 작성한다.
3. **변경 후 테스트 통과** — 정책 수정 후 기존 + 신규 테스트가 모두 통과하는지 확인한다.
4. **의도 명시** — 커밋 메시지에 정책 변경 사유를 명시한다.

## 테스트 패턴

```
// 경계값 테스트
test('만료일이 오늘이면 유효하다')
test('만료일이 어제면 무효하다')

// 상태 전이 테스트
test('승인 상태에서만 결제가 가능하다')
test('취소된 주문은 재승인할 수 없다')

// 금액 계산 테스트
test('할인율 적용 후 소수점 이하는 버린다')
test('최소 결제 금액 미만이면 에러를 반환한다')
```
