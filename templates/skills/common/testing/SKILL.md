## 테스트 작성 가이드

### 구조
- **Arrange-Act-Assert** 패턴을 따른다.
- 하나의 테스트에 하나의 개념만 검증한다.
- 테스트는 독립적이어야 한다 — 실행 순서에 의존하지 않는다.

### 네이밍
- 테스트 이름은 **무엇을 검증하는지** 명확히 드러내야 한다.
- `test_유효한_이메일이면_true를_반환한다` > `test1`

### 무엇을 테스트할 것인가
- 비즈니스 로직: 80%+ 커버리지
- 크리티컬 경로 (결제, 인증): 90%+ 커버리지
- 엣지 케이스: null, empty, 경계값, 에러 시나리오
- 100% 커버리지를 추구하지 않는다 — getter/setter 테스트는 불필요

### 하지 말 것
- `setTimeout` / `sleep`으로 대기하지 않는다 — 폴링, 이벤트, 테스트 클럭 사용
- 외부 서비스를 직접 호출하지 않는다 — mock/stub 사용
- 테스트에서 조건 분기(`if`)를 쓰지 않는다 — 별도 테스트로 분리
- 테스트 간 상태를 공유하지 않는다

### 실패 테스트 처리
- 실패 메시지는 무엇이 기대되었고 실제로 무엇이었는지 명확히 드러내야 한다.
- 실패한 테스트를 주석 처리하지 않는다 — 수정하거나 삭제한다.
