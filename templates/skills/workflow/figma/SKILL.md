## /figma 워크플로우

`/figma <URL>` 로 호출한다. Figma MCP가 연결되어 있어야 한다.

### 1단계: URL 파싱

- URL 패턴: `figma.com/design/:fileKey/...?node-id=:nodeId`
- nodeId의 `-`는 `:`로 변환 (예: `9248-29156` → `9248:29156`)
- 브랜치 URL(`/branch/:branchKey/`)이면 `branchKey`를 `fileKey`로 사용
- URL에 `node-id`가 없으면 `get_metadata`를 nodeId 없이 호출하여 페이지 목록을 먼저 확인한다

### 2단계: 정찰 — 메타데이터 수집 (병렬)

각 URL에 대해 **병렬**로 실행:

- `get_metadata` — 레이어 이름, 위치, 크기 파악 (경량)
- `get_screenshot` — 전체 레이아웃 시각 확인 (경량)

**중요: 이 단계에서는 절대 `get_design_context`를 호출하지 않는다.** 상위 Frame을 통째로 긁으면 토큰 소모가 폭발한다. 구조 파악 전에 드릴다운 대상을 확정한다.

### 3단계: 드릴다운 대상 선정

`get_metadata` 결과에서 **의미있는 최소 단위 노드**를 선별한다.

- 자동 생성 이름(`Rectangle`, `Vector`, `Group`, `Frame`)은 단독 후보로 올리지 않는다
- 의미있는 이름을 가진 컴포넌트/인스턴스 노드만 후보로 추출한다
- 전체 화면 분석이라도 **섹션 단위로 쪼갠 목록**을 먼저 정리한다

### 4단계: 상세 정보 수집 (필요한 경우에만)

확정된 최소 단위 노드들에 대해서만 `get_design_context` 호출:

- **기본: `excludeScreenshot: true`** — 토큰 절감
- **스크린샷 포함은 명시적 요청 시에만**: 사용자가 "스크린샷", "시각적으로", "어떻게 생겼" 등 요청한 경우
- **400 에러 시**: 하위 노드 단위로 **분할 호출**로 전환 (중단하지 않는다)

### 5단계: 출력

```
## Figma 분석 결과

### 컴포넌트 구조
| 컴포넌트 | 크기 | 상세 |
|---------|------|------|
| ...     | ...  | ...  |

### 레이아웃
- 전체 캔버스: WxH
- 정렬 방식: ...

### 디자인 토큰
- 색상: ...
- 간격: ...
- 폰트: ...
```

### Figma 링크 구분

- **기획 Figma (와이어프레임)** — 스펙/조건/플로우 추출
- **디자인 Figma (시안)** — UI 컴포넌트 구조, 색상, 간격, 폰트 추출
- 두 링크가 모두 있으면 둘 다 분석한다

### 토큰 절약 원칙

1. `get_metadata` + `get_screenshot`으로 **구조를 먼저 파악**한다
2. `get_design_context`는 **확정된 최소 단위 노드에만** 호출한다
3. 상위 Frame을 통째로 호출하지 않는다
4. `excludeScreenshot: true`가 기본값이다
5. 에러 시 분할 호출로 전환한다
