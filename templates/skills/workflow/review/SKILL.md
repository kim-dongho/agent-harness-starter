## /review 워크플로우

`/review` 로 호출한다. 현재 브랜치의 변경 사항을 체계적으로 리뷰한다.

### 1. 변경 파일 분석

```bash
git diff {{BASE_BRANCH}}...HEAD --stat
git diff {{BASE_BRANCH}}...HEAD
```

변경된 파일 목록과 diff를 확인한다.

### 2. 금지 패턴 체크

프로젝트 rules에 정의된 금지 패턴 위반을 검사한다:
- 하드코딩된 시크릿/크리덴셜
- 디버그 로그 (`console.log`, `print()`, `System.out.println` 등)
- 주석 처리된 코드 블록
- TODO/FIXME (이슈 번호 없이)
- 스택별 금지 패턴 (rules 디렉토리 참조)

### 3. 정책 검증

변경 범위에 정책 키워드가 포함되면:
- 해당 정책의 테스트 존재 여부를 확인한다
- 정책 변경 시 기존 테스트가 깨지지 않는지 확인한다
- 정책 변경 사유가 커밋 메시지에 명시되었는지 확인한다

### 4. 코드 품질 리뷰

code-review skill의 체크리스트를 적용한다:
- 정확성 (로직 에러, 엣지 케이스)
- 보안 (injection, 인증/인가)
- 성능 (N+1, 메모리 누수)
- 에러 처리 (무시된 에러, 리소스 정리)
- 유지보수성 (중복, 네이밍)

### 5. 리뷰 결과 출력

심각도별로 그룹화하여 제시한다:

```
## 리뷰 결과

### Critical (머지 차단)
- [파일:라인] 내용

### Important (논의 필요)
- [파일:라인] 내용

### Suggestion (비차단)
- [파일:라인] 내용

### 요약
- 변경 파일: N개
- 지적 사항: Critical N, Important N, Suggestion N
- 정책 관련 변경: 있음/없음
```
