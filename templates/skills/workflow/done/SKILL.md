## /done 워크플로우

`/done` 으로 호출한다. 아래 품질 게이트를 순서대로 통과한 후 커밋 → PR을 생성한다.

### 품질 게이트 (5단계)

모든 게이트를 통과해야 다음 단계로 진행한다. 실패 시 원인을 수정한 후 재시도한다.

#### Gate 1. 코드 품질
```bash
{{LINT_COMMAND}}
```
lint, type-check가 통과하는지 확인한다.

#### Gate 2. 테스트 (Self-healing Loop)
```bash
{{TEST_COMMAND}}
```
변경 사항과 관련된 테스트를 실행한다. 실패 시 아래 자가 수정 루프를 따른다:

1. 테스트 실행 → 실패 로그 확인
2. 에러 메시지를 분석하여 원인 파악
3. 코드 수정
4. 테스트 재실행
5. **최대 3회 반복** — 3회 실패 시 사용자에게 보고하고 중단

```
retry 1/3: ❌ expect(result).toBe(true) → 원인 분석 → 수정 → 재시도
retry 2/3: ❌ TypeError: cannot read property → 원인 분석 → 수정 → 재시도
retry 3/3: ✅ 통과
```

#### Gate 3. 정책 보호
변경된 파일에 정책 키워드(날짜 계산, 금액, 상태 전이, 필터 조건)가 포함되어 있으면:
- 해당 정책을 검증하는 테스트가 존재하는지 확인한다
- 테스트가 없으면 작성을 요구한다

#### Gate 4. 범위 검증
```bash
git diff --stat
```
의도하지 않은 파일 변경이 없는지 확인한다. 모든 변경 라인이 작업 요청에 직접 연결되는지 검토한다.

#### Gate 5. 컨벤션
- 커밋 메시지가 프로젝트 컨벤션을 따르는지 확인한다
- 불필요한 파일(.env, .DS_Store 등)이 포함되지 않았는지 확인한다

### 커밋 + MR 생성

5단계를 모두 통과하면:

1. **변경 내용 분석** — 변경 사항을 요약한다
2. **커밋** — 프로젝트 커밋 컨벤션에 맞는 메시지로 커밋한다
3. **Push** — 현재 브랜치를 remote에 push한다
4. **MR 생성**:
{{PR_CREATE_COMMAND}}
5. **이슈 상태 변경**:
{{ISSUE_DONE_COMMAND}}
